# 博客风格建议与文章初稿

## 一、适合 CSDN 的 4 种文章风格建议

### 风格 A：实战复盘型（推荐）
特点：
- 以“从 0 到 1 设计并落地一个 skill”为主线
- 兼顾过程、问题、决策与结果
- 最适合 CSDN 技术读者

优点：
- 真实感强
- 容易建立可信度
- 既能讲方法，也能讲工程落地

适合标题：
- 我把一个 SDD 协作方法做成了 OpenClaw Skill：从设计到落地复盘
- 从 0 到 1 设计 `sdd-orchestrator`：一个面向长期人机协作的 SDD Skill 实战复盘

---

### 风格 B：方法论拆解型
特点：
- 更强调 SDD v2 本身的设计思想
- 把 skill 作为方法落地载体来讲
- 适合想突出“为什么这样设计”的文章

优点：
- 理论框架感更强
- 容易形成方法论辨识度

风险：
- 如果写得太抽象，CSDN 读者会觉得不够落地

---

### 风格 C：工程落地指南型
特点：
- 更强调目录结构、SKILL.md、references、部署到 OpenClaw skills 目录、识别验证、git 管理
- 偏“教程 + 经验总结”

优点：
- 非常适合搜教程的人
- 搜索流量友好

风险：
- 容易把文章写得偏工具说明书，削弱这次协作过程本身的价值

---

### 风格 D：人机协作反思型
特点：
- 更强调“为什么单纯模板不够”“为什么要有状态卡、联动矩阵、handoff 协议”
- 从人与 AI 长期协作视角去讲

优点：
- 很有观点
- 容易形成个人风格

风险：
- 需要控制表达，不然会偏虚

---

## 我的建议

如果这篇是第一篇对外发布的 `sdd-orchestrator` 相关文章，**最推荐用：风格 A（实战复盘型）**。

原因：
1. 最适合 CSDN 读者阅读习惯
2. 能把方法、工程、落地、反思自然串起来
3. 不容易写空
4. 有利于后续再拆出方法论篇、教程篇、试运行篇

# 文章初稿（按“实战复盘型”写）

# 我把一个 SDD 协作方法做成了 OpenClaw Skill：`sdd-orchestrator` 从设计到落地复盘

## 前言

最近我在做一件对我来说挺重要的事：

> 把一套面向长期人机协作的 SDD（Spec-Driven Development）方法，真正收成一个可被 OpenClaw 识别和调用的 Skill。

这件事一开始看起来像是在“写几个模板”，但真做下去后很快发现：
问题远不只是 Spec / Plan / Tasks 三件套。

真正难的是这些：

- 什么时候该进入下一阶段？
- 草案和定稿怎么区分？
- 中断后靠什么恢复？
- 文档变了以后，其他相关文档要不要同步？
- Implement 阶段如果要交给 Coding Agent / 子 agent，怎么交接才不失控？

所以最后做出来的，不再只是一个模板集合，而是一个更像“协作编排器”的东西：`sdd-orchestrator`。

这篇文章，我想完整复盘一下这次从 0 到 1 的设计、收敛、工程化、部署与验证过程。

---

## 一、为什么要做 `sdd-orchestrator`

很多人提到 SDD，第一反应往往是：

- 写 spec
- 写 plan
- 写 tasks
- 然后开始实现

但如果你真的把它放进长期的人机协作里，会发现这里有很多隐含问题没有被解决。

### 1. 只有模板，不等于有流程
很多 SDD 实践提供了模板，但没有回答：

- 哪个阶段可以进入下一阶段
- 如果当前阶段还没确认，是否能继续写后面的文档
- checklist 和模板各自负责什么

结果就是：文档看起来写了很多，但流程其实是散的。

### 2. 只有当前对话，不等于有连续性
如果一次会话被打断，下一次重新打开时：

- 当前阶段是什么？
- 哪些文档已经确认？
- 下一步唯一推荐动作是什么？

如果这些信息只存在聊天记录里，而不在项目文档中，恢复成本会非常高。

### 3. Implement 阶段天然容易失控
当实现开始后，尤其是交给 Coding Agent / 子 agent 时，很容易出现：

- 越界实现
- 静默扩 scope
- 改了代码但没更新文档
- 完成了局部动作却没有阶段状态闭环

所以我越来越明确地意识到：

> 如果要把 SDD 真正用于长期协作，就不能只做模板，必须把“阶段门禁、状态锚点、文档联动、handoff 规则”也一并收进去。

这就是 `sdd-orchestrator` 的出发点。

---

## 二、我想做的不是“模板包”，而是“协作编排器”

在这次设计里，我最终给 `sdd-orchestrator` 的定位不是：

- 文档模板合集

而是：

- **SDD v2 协作编排器**

它负责的事情包括：

- 判断当前是否应该进入 SDD
- 判断当前处于哪个阶段
- 起草和收口阶段文档
- 维护 `sdd-status.md`
- 保持文档联动一致性
- 支撑中断恢复
- 在 Implement 前准备 handoff
- 把 Review 拉回到 Spec 与任务完成度上

这个定位变化很关键。

因为一旦把它理解成“编排器”，很多以前容易漏掉的问题就会自然浮出来。

---

## 三、这次设计中逐步收清的几个关键点

### 1. `sdd-status.md` 必须成为恢复主锚点
这是整套设计里我认为最关键的点之一。

如果一个项目中断后，下一次只能靠聊天记录恢复，那这套体系就不算真正稳定。

所以我明确把 `sdd-status.md` 提升成了：

- 当前阶段锚点
- 文档状态锚点
- 中断恢复锚点
- 下一步动作锚点

也就是说，恢复优先依赖项目文档，而不是依赖上下文记忆。

---

### 2. 文档状态和阶段状态必须分开
这个点如果不明确，后面非常容易混。

例如：
- `spec.md` 可能还是草案中
- 但需求讨论阶段已经结束
- 或者阶段已经切到 Plan 准备，但某个文档还待审核

所以我们明确区分了：

#### 文档状态
- 草案中
- 待审核
- 已确认
- 已替换
- 已废弃

#### 阶段状态
- 未开始
- 进行中
- 待收口
- 已完成
- 已回退

这个区分看起来小，实际上对恢复和联动都很重要。

---

### 3. 文档联动更新必须成为第一类规则
在很多实践里，代码变了以后文档常常是“有空再补”。

但如果要把 SDD 当协作系统来用，这种思路就会让体系很快失真。

所以这次我专门做了：
- `SDD 联动更新矩阵.md`

它的核心作用就是回答：

> 某个文档变了，哪些文档要同步检查和更新？

这让“联动更新”不再只是口头提醒，而是有显式规则可依。

---

### 4. Implement 阶段不能是黑箱
如果 Implement 一开始就变成“丢给 agent 然后等结果”，长期来看是不可控的。

所以我把实现前后的链路也一起收进来了：

- `execution-contract.md`
- `implement-handoff-checklist.md`
- `Coding Agent Handoff 协议.md`
- `实现层入口与链路.md`

这样做的目的，是让：

- 编排层
- 交接层
- 施工层

这三层边界清楚，而不是混成一团。

---

## 四、目录结构是怎么一步步收成现在这样的

一开始，`sdd-orchestrator` 更像一个工作区里的资产目录。

后来随着内容增多，我逐步把它收成了一个更像正式 Skill 的结构。

当前仓库结构大致是这样：

```text
sdd-orchestrator/
├── README.md
├── SKILL.md
└── references/
    ├── 索引与导航.md
    ├── SDD 模板总则.md
    ├── SDD 联动更新矩阵.md
    ├── Coding Agent Handoff 协议.md
    ├── 实现层入口与链路.md
    ├── sdd-orchestrator 首次试运行方案.md
    ├── core-templates/
    └── checklists/
```

这里面有几个关键收敛动作：

### 1. 主文件瘦身
`SKILL.md` 不再承担大量细节解释，而是更像：
- 触发入口
- 场景分流器
- 最小规则集
- 读取导航

### 2. references 下沉
详细规则、模板、清单、实现层配套全部放到 `references/`，避免主文件过重。

### 3. 补二级说明文件
我还分别给：
- `core-templates/`
- `checklists/`

补了极简目录说明，降低后续使用时的认知负担。

---

## 五、从“仓库”到“真正可用的 OpenClaw Skill”

这一步其实是我觉得最容易被忽略，但又非常重要的一步。

一开始虽然我已经有了：
- `SKILL.md`
- `references/`
- GitHub 仓库

但这还不等于 OpenClaw 已经能用它。

后来我确认到：

> OpenClaw 在工作区下确实有一个实际扫描的自定义 skills 目录：
> `/root/.openclaw/workspace/skills`

而且 `openclaw skills list` 里能看到很多 `openclaw-workspace` 来源的 skill。

于是我采取了一个更合理的策略：

### 源仓和运行时 skill 分离

#### 源仓
- `/root/.openclaw/workspace/sdd-orchestrator`
- 用于版本管理、README、迭代和 push GitHub

#### 运行时 skill
- `/root/.openclaw/workspace/skills/sdd-orchestrator`
- 只放运行时需要的内容：
  - `SKILL.md`
  - `references/`

为什么不把整个仓库都搬过去？

因为运行时 skill 应该尽量精简，README、`.git` 等文件都不必进入运行时目录。

最后验证结果也很直接：

```bash
openclaw skills info sdd-orchestrator
```

返回结果显示：
- `sdd-orchestrator ✓ Ready`
- `Source: openclaw-workspace`

这说明它已经不只是“一个设计好的仓库”，而是 **被 OpenClaw 真正识别的自定义 Skill**。

---

## 六、这次过程中我最有感触的几个经验

### 1. 先把骨架做对，再谈内容打磨
很多时候，文档写得不顺，不一定是表达问题，而是结构没收清。

这次真正有价值的动作，不是单纯改词，而是：
- 明确角色边界
- 明确导航层级
- 明确哪些该在主文件，哪些该下沉到 references

### 2. skill 仓库 ≠ 运行时 skill
这是这次很重要的工程认知。

有 GitHub 仓库，不代表 OpenClaw 已经能加载；
有 `SKILL.md`，也不代表系统已经自动识别。

真正要跑通，还得明确：
- skill 放在哪
- 系统扫描哪里
- 部署态和源仓如何分工

### 3. 中断恢复和联动更新，比模板本身更重要
如果只能“写一版好看的 spec”，但不能：
- 可靠恢复
- 可靠联动
- 可靠 handoff

那这套体系很难真正用于长期协作。

### 4. 真正的下一步不是继续空转式优化，而是试运行
做到现在这个阶段，我越来越确定：

> 一套方法什么时候算真的成型，不是看文档写到多漂亮，而是看它能不能进入真实项目并稳定跑一轮。

所以 `sdd-orchestrator` 当前最合理的状态，不是“永远继续打磨”，而是：

- 文档层面收口为可试运行定稿
- 然后进入真实项目试运行
- 再根据真实摩擦点迭代

## 结语

这次做 `sdd-orchestrator`，对我来说最大的收获不是“写出了一套模板”，而是更清楚地意识到：

> 长期人机协作真正需要的，不只是文档格式，而是一整套能支撑推进、恢复、联动和交接的协作编排机制。

从这个意义上说，`sdd-orchestrator` 现在才刚刚从“方法想法”变成“可运行系统”的雏形。

后面真正有价值的，不是继续抽象，而是把它放进真实项目里，让它接受真实摩擦的检验。

只有那样，它才会真正长成一个靠谱的 Skill。
