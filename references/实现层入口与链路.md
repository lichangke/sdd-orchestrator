# 实现层入口与链路

## 0. 文档定位

本文件用于说明 `sdd-orchestrator` 与实现层的配套关系。  
它的职责是：

1. 说明 `sdd-orchestrator` 负责什么
2. 说明实现层负责什么
3. 说明 `Coding Agent Handoff 协议.md` 在体系中的位置
4. 说明什么时候从编排层切到实现层
5. 说明什么时候必须从实现层回到主对话

默认原则：
- `sdd-orchestrator` 负责推进
- handoff 协议负责交接
- 实现层负责施工
- `sdd-status.md` 负责连续性

---

## 1. 总体分工

### 1.1 `sdd-orchestrator` 负责什么
- 需求澄清
- 阶段识别
- 阶段门禁
- 文档起草与收口
- `sdd-status.md` 维护
- 文档联动更新
- 中断恢复
- Implement 前 handoff 编排
- Review / 验收组织

一句话：
> `sdd-orchestrator` 负责“怎么推进”。

---

### 1.2 实现层负责什么
- 阅读代码上下文
- 按 task / task group 执行实现
- 跑验证
- 修复问题
- 留下执行痕迹
- 更新任务状态与状态卡
- 在阻塞或阶段完成时回报

一句话：
> 实现层负责“怎么施工”。

---

### 1.3 `Coding Agent Handoff 协议.md` 负责什么
- 定义 handoff 前置条件
- 定义 handoff 最小上下文包
- 定义 Coding Agent 权限边界
- 定义汇报规则与阻塞规则
- 定义实现期文档更新纪律
- 定义完成定义与回退规则

一句话：
> handoff 协议负责“怎么把施工任务稳地交出去”。

---

## 2. 什么时候切到实现层

默认在以下条件同时满足时，进入实现层：

- `spec.md` 已确认
- `plan.md` 已确认
- `tasks.md` 已确认
- `execution-contract.md` 已形成可执行版本
- `sdd-status.md` 当前阶段与下一步动作清楚
- 当前要执行的 task / task group 范围清楚

若上述条件不满足，默认继续停留在编排层。

---

## 3. 进入实现层前的最小读取顺序

进入 Implement 前，建议按以下顺序读取：

1. `core-templates/execution-contract.md`
2. `checklists/implement-handoff-checklist.md`
3. `Coding Agent Handoff 协议.md`
4. `core-templates/sdd-status.md`

必要时再补：
- `core-templates/tasks.md`
- `core-templates/plan.md`
- `core-templates/spec.md`

---

## 4. 从主对话到实现层的默认链路

推荐链路如下：

1. 用户与主助手完成高层讨论
2. `sdd-orchestrator` 推进并确认上游阶段文档
3. 使用 `implement-handoff-checklist.md` 做交接前检查
4. 使用 `Coding Agent Handoff 协议.md` 组织 handoff 内容
5. 切换到 Coding Agent / 子 agent 执行实现
6. 实现完成或阻塞后，再回到主对话

---

## 5. 什么时候必须回到主对话

以下情况必须从实现层回到主对话：

1. 关键前提不成立
2. 需要修改 Plan 级方案
3. 需要扩大本轮范围
4. 当前 task 定义不足以继续执行
5. 存在多个合理方案，需要用户拍板
6. review 结论需要正式收口

默认原则：
> 需要边界决策、方案决策、范围决策时，回到主对话；不在实现层私自拍板。

---

## 6. 什么时候不必回到主对话

以下情况默认可在实现层内部闭环处理：

- 小范围必要调整
- 局部合理重构
- 普通调试
- 小问题修复
- 既有边界内的验证与补修

前提是：
- 不改变已确认 Spec
- 不扩大范围
- 不破坏当前执行协议
- 按要求同步状态与痕迹

---

## 7. 当前实现层文件关系

当前实现层相关文件为：
- `core-templates/execution-contract.md`
- `checklists/implement-handoff-checklist.md`
- `Coding Agent Handoff 协议.md`
- `实现层入口与链路.md`
- `core-templates/sdd-status.md`

关系如下：
- `execution-contract.md`：定义本轮实现规则
- `implement-handoff-checklist.md`：定义是否具备进入 Implement 的条件
- `Coding Agent Handoff 协议.md`：定义如何正式交接给 Coding Agent
- `实现层入口与链路.md`：定义编排层与实现层的切换说明与 handoff 链路总览
- `sdd-status.md`：定义实现推进中的状态锚点

---

## 8. 未来扩展边界

当前实现层通过 handoff 协议与现有模板体系配合即可运行。  
后续若新增实现层 skill，应保持以下分工不变：

- 编排层：`sdd-orchestrator`
- 交接层：handoff 协议 / handoff 检查
- 施工层：Coding Agent / 实现类 skill

不要把三层重新混成单层设计。

---

## 9. 当前版本结论

本文件作为实现层入口文件，用于帮助：
- 让编排层与实现层职责更清楚
- 让 handoff 相关文件形成清晰链路
- 避免“有执行协议，但不知道该先读什么、怎么切换”的问题

一句话总结：

> `sdd-orchestrator` 管推进，handoff 协议管交接，实现层管施工，`sdd-status.md` 管连续性。
