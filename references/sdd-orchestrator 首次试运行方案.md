# 《sdd-orchestrator 首次试运行方案》

## 0. 文档目的

本方案用于为 `sdd-orchestrator` 的第一次真实使用提供一个低风险、可观察、可复盘的试运行框架。  
目标不是一上来追求“完美落地”，而是通过一轮真实项目推进，验证这套 skill 的：

1. 路由是否顺畅
2. 阶段门禁是否清晰
3. 模板是否够用
4. checklist 是否真正有帮助
5. Implement handoff 链路是否成立
6. 中断恢复与联动更新机制是否可靠

本方案默认服务于：
- `sdd-orchestrator/SKILL.md`
- `sdd-orchestrator/references/`
- 后续真实项目试运行与反馈回灌

---

## 1. 试运行目标

首次试运行的目标不是“把项目做到多完整”，而是验证这套 skill 包是否已经足够支撑一轮真实协作。

本轮重点验证以下问题：

### 1.1 编排层是否可用
- 主入口是否够清晰
- 场景分流是否自然
- `SKILL.md` 是否足够轻而不失效

### 1.2 文档体系是否可用
- `spec.md` / `plan.md` / `tasks.md` 是否真正能承载推进
- `sdd-status.md` 是否真能作为恢复锚点
- 主模板与 checklist 的分工是否清楚

### 1.3 实现层链路是否可用
- `execution-contract.md` 是否足够形成 handoff 前执行规则
- `implement-handoff-checklist.md` 是否能判断“是否可交接”
- `Coding Agent Handoff 协议.md` 是否足够支撑真实交接

### 1.4 联动与恢复机制是否可用
- 文档变化后，联动更新是否容易漏
- 中断后是否真的能仅靠文档恢复
- 文档状态 / 阶段状态是否仍容易混淆

---

## 2. 首次试运行的选择原则

首次试运行不应选择过大、过散、过重依赖外部变量的项目。  
应优先选择：

### 2.1 边界相对清楚的项目
避免一开始就因为需求极不稳定而把试运行问题和项目本身问题混在一起。

### 2.2 有真实推进价值的项目
不要拿完全虚构练习题；最好是后续本来就要推进的真实项目。

### 2.3 能覆盖 SDD 核心链路的项目
至少应能覆盖：
- 需求讨论
- Specify
- Plan
- Tasks
- Execution Contract / Implement 准备

如果条件允许，再覆盖：
- Implement handoff
- Review

---

## 3. 当前最适合的试运行对象

结合现有长期主线，首次试运行最适合的对象是：

## `local-kb-assistant`

原因如下：

### 3.1 它本来就是当前 AI 项目主线优先级第一
所以试运行不会脱离长期主线。

### 3.2 它边界已经相对容易收清
最小闭环已经比较明确：
- 本地文档导入
- 索引建立
- 提问回答
- 引用出处返回

### 3.3 它适合覆盖 SDD 核心链路
这个项目足够真实，也足够有结构，适合验证：
- spec 是否能写清边界
- plan 是否能收清技术路径
- tasks 是否能拆出合理 task group
- execution contract 是否能为 implement 做准备

### 3.4 它不过分复杂
第一次试运行不宜选高度复杂项目；`local-kb-assistant` 的复杂度更适合用来试运行方法论。

---

## 4. 首轮试运行边界建议

首次试运行不建议一口气跑完整个长期项目。  
更适合采用：

## “跑完整条方法链，但只覆盖项目的第一版最小闭环”

也就是：

### 本轮建议覆盖到：
1. 需求讨论
2. Specify
3. Plan
4. Tasks
5. Execution Contract
6. Implement handoff 准备

### 可选扩展：
7. 真实 handoff 给 Coding Agent
8. 完成一个最小 task group 的实现
9. 做一次 Review

### 4.1 为什么不建议第一次就追求全量实现
因为第一次试运行的核心任务是验证 **skill 体系本身**，  
不是让某个项目尽快完工。

如果第一轮就把目标设成“整项目做完”，容易把：
- 方法论问题
- 项目难度问题
- 实现细节问题

混成一团。

---

## 5. 首次试运行的推荐节奏

推荐分三段进行。

### 第一段：编排层验证
目标：
- 看 `SKILL.md` 是否足够引导推进
- 看 spec / plan / tasks 模板是否好用
- 看 checklist 是否真能发挥门禁作用

成功标志：
- 至少形成一版明确的 `spec.md`
- 至少形成一版清晰的 `plan.md`
- 至少形成一版可执行的 `tasks.md`
- `sdd-status.md` 可正确表达当前阶段与下一步动作

### 第二段：实现前验证
目标：
- 看 `execution-contract.md` 是否够支撑 Implement
- 看 handoff checklist 是否能判断是否具备交接条件
- 看实现层入口说明和 handoff 协议是否衔接顺畅

成功标志：
- 形成可执行的 `execution-contract.md`
- 明确一个 task group 可进入 handoff
- handoff 上下文包可以被稳定组织出来

### 第三段：最小实现验证（可选但强烈建议）
目标：
- 真正验证“编排层 → 交接层 → 施工层 → 回到主对话”的链路是否成立

成功标志：
- 至少完成一个最小 task group 的 handoff
- 实现层能按规则执行
- 执行结果能回写到 `sdd-status.md` / task 状态
- 主对话能根据结果继续推进或收口

---

## 6. 本轮重点观察项

首次试运行时，不要只看“有没有推进”，要重点观察以下摩擦点。

### 6.1 主入口摩擦
- `SKILL.md` 是否仍然过重
- 场景分流是否真的足够自然
- 使用时是否仍需要大量额外解释

### 6.2 模板摩擦
- 模板是否过重或过轻
- 哪些字段总是空着
- 哪些信息总被重复写
- 哪些内容本该在别处记录

### 6.3 checklist 摩擦
- checklist 是否真的能判断是否可进入下一阶段
- 是否出现 checklist 变成重复模板的问题
- 是否出现“看完 checklist 仍不知道能不能进”的情况

### 6.4 状态摩擦
- `sdd-status.md` 是否足够表达恢复信息
- 文档状态 / 阶段状态是否仍容易混淆
- 中断恢复时是否还需要依赖聊天上下文

### 6.5 联动摩擦
- 文档一变，是否容易漏改别处
- 联动更新矩阵是否足够实用
- 实现推进后，是否容易忘记更新任务或状态

### 6.6 handoff 摩擦
- `execution-contract.md` 是否足够具体
- handoff 协议是否足够落地
- Coding Agent 是否会越界、静默乱改或回报不足

---

## 7. 本轮不重点追求的事情

为了避免目标失焦，首次试运行不把以下内容作为首要目标：

- 文档写得极致漂亮
- 所有模板一次定终版
- 整个项目一轮做完
- 一开始就覆盖所有复杂异常场景
- 立即形成可发布 skill

本轮优先级始终是：

> **验证这套 skill 是否能真实跑通。**

---

## 8. 试运行后的输出要求

每次试运行结束后，至少应产出一份简短复盘，记录：

### 8.1 跑通了什么
- 哪些阶段顺利
- 哪些文档够用
- 哪些链路已验证成立

### 8.2 卡住了什么
- 卡在编排层、模板层、checklist 层，还是实现层
- 是规则问题、文档问题，还是项目本身问题

### 8.3 建议如何修改
- 修改 `SKILL.md`
- 修改某个模板
- 修改某个 checklist
- 修改导航 / 入口
- 修改 handoff 协议

### 8.4 变更优先级
- P1：不改会影响继续试跑
- P2：建议优化但不阻塞
- P3：观察更多样本后再定

---

## 9. 本轮试运行后的迭代原则

首次试运行结束后，不建议立刻大修所有文档。  
应按以下原则迭代：

### 9.1 只改真实暴露的问题
避免把“可能会有问题”当成必须修改的问题。

### 9.2 优先修阻塞继续试运行的问题
先保证第二轮还能顺利跑。

### 9.3 小步迭代
一次只改一组相关问题，避免大范围重构文档体系。

### 9.4 保留稳定骨架
不要因为局部摩擦就推翻整个结构；优先在现有骨架上微调。

---

## 10. 当前建议结论

`sdd-orchestrator` 当前已经达到“可试运行定稿”状态。  
首次试运行建议：

- 以 `local-kb-assistant` 作为第一试运行对象
- 先覆盖从需求讨论到 Implement handoff 准备的完整链路
- 条件允许时，再做一个最小 task group 的真实 handoff 与实现验证
- 试运行后输出简短复盘，并只基于真实摩擦点迭代 skill

---

## 11. 一句话总结

> `sdd-orchestrator` 的首次试运行，不是为了证明文档写得多漂亮，而是为了验证：这套编排、模板、清单、状态、handoff 机制，能不能在真实项目里稳定跑通。
