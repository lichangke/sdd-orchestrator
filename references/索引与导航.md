# 索引与导航

## 0. 文档定位

本文件是 `sdd-orchestrator/references/` 的总入口。  
它的职责是：

1. 说明 references 目录内有哪些类别的文件
2. 说明每类文件分别负责什么
3. 说明不同任务场景下应优先读取哪些文件
4. 作为 `SKILL.md` 之后的第二入口文件，承接更细的内部导航

默认原则：
- `SKILL.md` 负责触发、总路由、最小规则
- 本文件负责 references 内部导航与读取顺序
- 具体规则、模板、清单、协议细节继续下沉到对应文件

---

## 1. 当前 references 结构总览

当前 references 目录分为五类：

### A. 上位规则层
- `SDD 模板总则.md`
- `SDD 联动更新矩阵.md`

作用：
- 规定模板体系的上位约束
- 规定文档联动更新纪律
- 规定文档状态 / 阶段状态 / 模板职责边界

---

### B. 导航与入口层
- `索引与导航.md`
- `实现层入口与链路.md`

作用：
- 承担 references 的总导航
- 承担实现层与 handoff 链路说明
- 帮助从编排层平稳切换到实现层

---

### C. 核心模板层
目录：`core-templates/`
- `spec.md`
- `plan.md`
- `tasks.md`
- `execution-contract.md`
- `sdd-status.md`
- `review.md`

作用：
- 承载 SDD 各阶段正式文档模板

---

### D. 检查清单层
目录：`checklists/`
- `specify-checklist.md`
- `plan-checklist.md`
- `tasks-checklist.md`
- `implement-handoff-checklist.md`
- `review-checklist.md`

作用：
- 对各阶段主模板执行门禁检查与收口检查

---

### E. 协议层
- `Coding Agent Handoff 协议.md`

作用：
- 规定主助手如何把 Implement 阶段任务交给 Coding Agent / 子 agent
- 规定 handoff 前置条件、最小上下文包、权限边界、汇报与状态更新纪律

## 2. 文件角色导航

### 2.1 先读哪些文件取决于你在做什么

#### 如果你在理解整套体系
按顺序读：
1. 本文件
2. `SDD 模板总则.md`
3. `SDD 联动更新矩阵.md`

#### 如果你在推进某个阶段
按顺序读：
1. 当前阶段主模板
2. 当前阶段 checklist
3. `core-templates/sdd-status.md`

默认提醒：
- 若当前阶段主文档尚未正式落盘到项目中，说明还处于讨论中
- 此时应先落盘当前阶段主文档，再做该阶段内容推进、门禁判断或下游推进
- `sdd-status.md` 也必须锚定这一真实文档状态，而不是先按对话理解切换阶段

#### 如果你在处理恢复或联动问题
按顺序读：
1. `core-templates/sdd-status.md`
2. `SDD 联动更新矩阵.md`
3. 当前阶段主模板
4. 对应 checklist

若你正在做阶段切换，优先查看：
- `SDD 联动更新矩阵.md` 中的“4.0 通用阶段切换动作清单”
- 再看对应具体阶段切换条目

#### 如果你在进入 Implement / handoff 给 Coding Agent
按顺序读：
1. `core-templates/execution-contract.md`
2. `checklists/implement-handoff-checklist.md`
3. `Coding Agent Handoff 协议.md`
4. `实现层入口与链路.md`
5. `core-templates/sdd-status.md`

#### 如果你在做 Review / 验收
按顺序读：
1. `core-templates/review.md`
2. `checklists/review-checklist.md`
3. `core-templates/spec.md`
4. `core-templates/tasks.md`
5. `core-templates/sdd-status.md`

## 3. 各文件的最小职责

### 上位规则
#### `SDD 模板总则.md`
回答：
- 模板体系怎么分层
- 主模板与 checklist 各负责什么
- 文档状态与阶段状态如何区分
- 项目级 / 功能级如何复用

#### `SDD 联动更新矩阵.md`
回答：
- 哪个文档变了以后要联动检查哪些文档
- 状态更新与内容更新的联动强度差异是什么
- Implement 阶段如何保证文档同步

---

### 导航与入口
#### `索引与导航.md`
回答：
- references 内部怎么读
- 哪些文件是规则，哪些是模板，哪些是清单，哪些是协议

#### `实现层入口与链路.md`
回答：
- `sdd-orchestrator` 和实现层如何分工
- Implement 为什么要切到实现层
- 什么时候必须回到主对话
- handoff 相关文件之间如何连起来

---

### 核心模板
#### `spec.md`
定义需求、用户故事、FR、边界、成功标准

#### `plan.md`
定义方案、技术上下文、模块划分、结构决策、MVP 路径

#### `tasks.md`
定义按用户故事组织的任务拆解、依赖关系与执行策略

#### `execution-contract.md`
定义本轮实现边界、汇报规则、阻塞规则、验证要求

#### `sdd-status.md`
定义当前阶段、文档状态、中断点、下一步动作，是恢复主锚点

#### `review.md`
定义 review 范围、Spec 满足度、验证结果、代码质量与结论

---

### 检查清单
每份 checklist 只回答一个问题：
- 当前阶段主文档够不够进入下一步

不要让 checklist 替代主模板写正式内容。

---

### 协议层
#### `Coding Agent Handoff 协议.md`
定义：
- 何时允许 handoff
- handoff 最小上下文包
- Coding Agent 权限边界
- 汇报规则与回退规则
- 实现阶段文档更新纪律

---

## 4. references 的使用顺序建议

### 4.1 默认读取顺序
当不确定该先读什么时，默认按这个顺序：

1. 先看本文件，确定当前属于哪类任务
2. 再读上位规则文件（若涉及规则判断）
3. 再读当前阶段主模板
4. 再读当前阶段 checklist
5. 若进入 Implement，则转入协议层与实现层入口文件

---

### 4.2 默认不建议的做法
- 不要一开始把 references 全读完
- 不要跳过 `sdd-status.md` 直接靠记忆恢复
- 不要先看 checklist 再忽略主模板
- 不要在进入实现层前跳过 `execution-contract.md` 与 handoff 协议

---

## 5. 与 `SKILL.md` 的职责边界

### `SKILL.md` 负责
- 触发描述
- 场景分流
- 最小规则
- 读取哪些文件

### 本文件负责
- references 内部结构总览
- 文件类别划分
- 更细的读取顺序
- 目录内部导航

默认原则：
> 不把过多目录说明塞回 `SKILL.md`；目录内部导航优先放在本文件。

---

## 6. 当前实现层配套关系

当前实现层相关文件为：
- `core-templates/execution-contract.md`
- `checklists/implement-handoff-checklist.md`
- `Coding Agent Handoff 协议.md`
- `实现层入口与链路.md`
- `core-templates/sdd-status.md`

默认链路如下：

1. `sdd-orchestrator` 负责推进到 Implement 前
2. `execution-contract.md` 规定本轮执行边界
3. `implement-handoff-checklist.md` 检查是否具备 handoff 条件
4. `Coding Agent Handoff 协议.md` 规定如何正式交接给 Coding Agent
5. `实现层入口与链路.md` 说明编排层与实现层如何切换
6. `sdd-status.md` 负责实现期连续性

---

## 7. 当前版本结论

本文件作为 references 入口文件，目的是降低 references 层的认知负担，避免：
- 已有很多文件，但不知道先读什么
- `SKILL.md` 被迫承担过多目录导航职责
- 编排层、模板层、清单层、协议层混在一起

一句话总结：

> `SKILL.md` 管路由，本文件管目录导航；规则看总则与矩阵，推进看模板与 checklist，施工切到实现层入口与 handoff 协议。
