# 《Coding Agent Handoff 协议（初稿）》

## 0. 文档目的

本协议用于规范主助手向 Coding Agent 交接实现任务时的默认方式。  
它解决的是：

1. 什么前提下可以把任务交给 Coding Agent  
2. 交接时必须提供哪些上下文  
3. Coding Agent 的权限边界是什么  
4. Coding Agent 该如何汇报、留痕、更新状态  
5. 出现阻塞或歧义时如何回到主对话  
6. 如何保证实现阶段与 SDD 文档体系保持一致

本协议默认服务于：
- `sdd-orchestrator`
- `execution-contract.md`
- `tasks.md`
- `sdd-status.md`

---

## 1. 角色分工

### 1.1 主助手（莫邪）
负责：
- 判断当前是否满足 handoff 条件
- 整理 handoff 所需上下文
- 明确任务边界、执行规则、汇报要求
- 在关键节点接收子 agent 回报
- 需要时把阻塞重新带回用户主对话

### 1.2 Coding Agent
负责：
- 读取交接上下文
- 按指定 task / task group 执行实现
- 运行必要验证
- 更新执行痕迹与状态
- 在关键阻塞或阶段完成时回报

### 1.3 用户
负责：
- 审核上游阶段产物
- 拍板边界、方案与阻塞决策
- 最终 review / 验收

---

## 2. 何时允许 handoff

只有在满足以下条件时，才允许把任务正式交给 Coding Agent：

- `spec.md` 已确认
- `plan.md` 已确认
- `tasks.md` 已确认
- `execution-contract.md` 已形成可执行版本
- `sdd-status.md` 当前阶段、当前中断点、下一步动作已清楚
- 当前要交接的 task / task group 范围已明确

如果上述任一项不满足，默认不直接 handoff，而是先回到对应阶段补齐。

---

## 3. handoff 最小上下文包

每次交给 Coding Agent 时，至少应提供以下信息。

### 3.1 任务身份信息
- 项目名称
- 功能名称（如有）
- 当前阶段：Implement
- 当前 task / task group 名称
- 当前优先级（如 P1 / P2 / P3）

### 3.2 目标与边界
- 本轮要完成什么
- 本轮明确不做什么
- 对应哪些用户故事 / FR
- 允许的修改范围
- 不允许越界的边界

### 3.3 上游依据
至少明确指向并要求阅读：
- `spec.md`
- `plan.md`
- `tasks.md`
- `execution-contract.md`
- `sdd-status.md`

必要时再补：
- `review.md` 草稿
- 相关 change-note
- 相关代码路径说明

### 3.4 执行规则
至少说明：
- 默认静默执行还是阶段性汇报
- 什么情况下必须停止并上报
- 是否允许小范围必要调整
- 是否允许局部重构
- 什么叫完成
- 完成后必须留下哪些痕迹

### 3.5 验证要求
至少明确：
- 主链路如何验证
- 当前 task group 的完成验证方式
- 若验证失败该如何处理
- “无验证结果不算完成”

---

## 4. Coding Agent 权限边界

### 4.1 默认允许
- 按已确认 tasks 执行
- 做小范围必要调整
- 补充极小型必要任务（需记录）
- 在不改变已确认边界前提下做合理局部重构
- 运行必要验证
- 修复实现中直接暴露的小问题

### 4.2 默认不允许
- 擅自修改已确认 Spec
- 擅自扩大范围
- 把一个 task 顺手做成大功能扩展
- 因个人偏好引入大重构
- 跳过验证直接宣称完成
- 绕过 `sdd-status.md` 与任务状态更新
- 用“我觉得更合理”替代用户已确认边界

### 4.3 必须停止并上报的情况
出现以下任一情况时，必须停止默认推进并上报：

1. 关键前提不成立
2. 需要修改 Plan 级方案
3. 当前 task 定义不足以支撑继续执行
4. 已确认需求与现实实现路径冲突
5. 需要扩大本轮范围
6. 存在多个合理方案且需要用户拍板
7. 发现实现代价远超预期，可能影响阶段策略

---

## 5. 汇报规则

### 5.1 默认汇报粒度
默认不汇报局部实现动作。  
只在以下情况汇报：

1. 完成一个大 task group
2. 出现关键阻塞
3. 出现风险偏移
4. 实现阶段整体完成

### 5.2 默认不汇报的内容
- 普通调试过程
- 小型尝试失败
- 局部代码修改细节
- 单条命令执行过程
- 小范围命名或格式调整

### 5.3 阻塞汇报最小格式
当必须上报时，至少应包含：

- 当前卡点是什么
- 为什么卡住
- 已尝试了什么
- 有哪些可选方案
- 推荐方案是什么
- 当前建议回退到哪个阶段（如需要）

### 5.4 完成汇报最小格式
当完成一个 task group 或本轮实现时，至少应包含：

- 已完成的任务范围
- 实际修改了什么
- 验证结果如何
- 当前文档状态已更新到什么程度
- 下一步建议动作是什么

---

## 6. 文档与状态更新纪律

这是 handoff 协议里最关键的一条之一。

### 6.1 每完成一个大 task group，至少同步更新
- `sdd-status.md`
- 当前任务状态
- 当前验证结果摘要
- 下一步动作
- 当前阻塞（如有）

### 6.2 如果实现导致边界、任务、执行规则变化
必须额外检查并必要时更新：
- `tasks.md`
- `execution-contract.md`
- `review.md` 范围
- 必要时回退 `plan.md` 或 `spec.md`

### 6.3 默认原则
> 代码推进不等于 handoff 完成；只有代码、状态卡、任务状态、验证结果形成一致闭环，才算本轮 handoff 任务完成。

---

## 7. 完成定义

Coding Agent 承接的任务，只有同时满足以下条件，才算完成：

- 已覆盖约定 task 范围
- 已形成真实代码改动
- 已完成必要验证
- 已更新 `sdd-status.md`
- 已同步任务状态
- 无未披露关键阻塞
- 已给出清晰的下一步动作建议

---

## 8. 回退规则

如果实现期发现以下问题，不应硬推，而应回退：

- Spec 理解存在关键歧义
- Plan 路径明显不可行
- Tasks 拆解不足以继续执行
- 当前执行范围已经明显漂移
- 原定验证方式不再成立

### 8.1 回退目标
根据问题级别，回退到：

- Specify
- Plan
- Tasks
- Execution Contract

### 8.2 回退时必须说明
- 为什么要回退
- 当前依据哪里失效了
- 哪个文档需要先修正
- 修正后才能如何继续

---

## 9. 标准 handoff 模板（简版）

下面是一份可直接复用的 handoff 指令骨架。

### Coding Agent Handoff 模板（简版）

你正在执行 [项目名 / 功能名] 的 Implement 阶段任务。  
当前目标是完成：[task group 名称 / task 名称]

#### 先阅读以下文档
- `spec.md`
- `plan.md`
- `tasks.md`
- `execution-contract.md`
- `sdd-status.md`

#### 本轮必须完成
- [...]
- [...]
- [...]

#### 本轮明确不做
- [...]
- [...]
- [...]

#### 对应依据
- 用户故事：
- FR：
- 任务组：

#### 权限边界
- 允许：小范围必要调整、局部合理重构、必要验证
- 不允许：修改 Spec、扩大范围、大重构、跳过验证

#### 汇报规则
默认静默执行，仅在以下情况汇报：
- 完成一个大 task group
- 出现关键阻塞
- 出现风险偏移
- 整体完成

#### 文档更新要求
完成后至少同步：
- `sdd-status.md`
- 当前任务状态
- 当前验证结果
- 下一步建议动作

#### 若遇阻塞
必须说明：
- 当前卡点
- 已尝试动作
- 可选方案
- 推荐方案
- 是否建议回退阶段

## 10. 与 `sdd-orchestrator` 的关系

本协议不是独立替代 `sdd-orchestrator` 的体系，  
而是 `sdd-orchestrator` 在 Implement 阶段的执行延伸。

关系如下：

- `sdd-orchestrator` 负责：阶段编排、阶段门禁、文档体系、恢复逻辑
- `execution-contract.md` 负责：本轮执行规则
- 本 handoff 协议负责：把执行规则稳定地下发给 Coding Agent
- Coding Agent 负责：实际施工与验证

---

## 11. 当前版本结论

本协议作为当前 Coding Agent handoff 的初稿，  
目标不是把所有实现行为写死，而是确保以下几件事稳定成立：

1. handoff 前置条件清楚
2. handoff 内容最小而充分
3. Coding Agent 权限边界清楚
4. 汇报与阻塞规则清楚
5. 状态更新与文档联动有明确纪律
6. 出问题时能稳定回到主对话与 SDD 阶段体系
